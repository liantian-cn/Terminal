# Terminal 开发规则

## 基本规则

- Windows 下默认用 PowerShell
- Python 命令统一用 `uv run`
- Python 目标版本是 3.12
- 修改前先看 `git status --short`
- 如果工作区已有未提交改动，先做一次备份提交
- 每次开发任务开始前，必须先生成一份 plan，再进入实现
- plan 统一放在 `docs/plans/`
- plan 文件名统一用 `YYYY-MM-DD-主题.md`
- 如果同一任务还有设计稿，设计稿文件名统一用 `YYYY-MM-DD-主题-design.md`
- plan 至少写清楚目标、改动范围、实现步骤、验证方式

## 代码组织

- 采集、解码、识别、状态还原、GUI、执行分层放置
- 模块名尽量写完整，不要图省事缩写
- 测试放在 `Terminal/tests/`
- `Terminal/notes/` 是实验笔记区，只放性能比较、API 验证、临时方案对比这类 Python 脚本
- 正式开发、重构、补测试、整理模块时，忽略 `Terminal/notes/`，不要把它当正式实现来源
- `Terminal/notes/` 下的脚本可以单独运行，但正式模块不要依赖这里的文件

## PySide6 界面代码风格

- 这一节只约束 `Terminal` 里的 PySide6 界面代码，例如 `Terminal/terminal/ui/` 下的窗口、页面、tab 和相关控件组装代码
- 风格参考 Qt for Python / PySide6 官方教学文档示例，用 MCP 查到的官方示例作为第一参考，不自己发明一套“更高级”的 GUI 写法
- 界面代码优先写得更傻瓜、更易懂，让人一眼看出这个页面在做什么，不要为了显得专业就多包一层抽象
- 搭界面时，优先按 `创建控件 -> 设置属性 -> 放进布局 -> 连接信号` 这种顺序展开，尽量让阅读顺序和界面生成顺序一致
- 方法名和变量名尽量直白，优先让人一看就知道用途，例如 `_build_ui`、`_build_form`、`_connect_signals`，不要堆难猜的缩写
- 如果一个页面、分组或控件区块的作用不够一眼看懂，就补足中文注释，先讲清楚“这块给谁用”“这块控制什么”“改这里会影响什么”
- 信号和槽函数尽量写在相关控件附近，连接关系要容易顺着看，不要把本来很短的交互拆成过度分散的跳转链
- 除非确实复用价值很高，否则不要为了样式统一、封装感或架构感，过早抽出很薄的基类、包装器和中间层

## 开发边界

- 不在 `Terminal` 任务里顺手改 `DejaVu`
- 不把共享协议说明散写到 Python 模块里，统一回 `Terminal/.context/` 里的协议文档
- 解码逻辑依赖的是共享协议，不是对游戏状态的猜测

## 推荐开发顺序

1. 先锁定矩阵区域
2. 再做切块和基础颜色识别
3. 再做图标、文字和状态还原
4. 最后才考虑 GUI 与策略调用

## 验证

- 基础测试命令: `uv run pytest`
- 缩小范围时可以只跑单测文件
- 对图像解码功能，优先补 fixture 和回归测试
